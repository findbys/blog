# uni-app 多端开发避坑指南

uni-app 的优势是多端复用，但多端不等于完全一致。真实项目里，平台差异、组件兼容、生命周期和发布链路都会影响开发效率。

## 样式差异

不同端对 CSS 的支持不完全一致。建议避免过于复杂的选择器和实验性属性，核心布局优先使用 flex，并在关键页面进行真机验证。

```css
.page {
  display: flex;
  flex-direction: column;
  min-height: 100vh;
}
```

## 条件编译

平台专属逻辑使用条件编译隔离，避免把兼容判断散落在业务代码中。

```ts
// #ifdef MP-WEIXIN
console.log('微信小程序逻辑')
// #endif

// #ifdef H5
console.log('H5 逻辑')
// #endif
```

## 生命周期差异

页面生命周期和组件生命周期要分清。页面级数据刷新通常放在 `onShow`，组件内部状态初始化则放在组件生命周期里。

## 请求与鉴权

建议统一封装请求层，集中处理 baseURL、token、错误提示和登录失效。多端环境变量也应收口到一个配置模块。

## 发布检查

每次发布前至少检查三类问题：关键页面真机表现、登录支付等平台能力、分包和资源体积。不要只在 H5 调通后就直接发布小程序或 App。

## 总结

uni-app 的关键是“共享业务，隔离差异”。把平台差异控制在清晰边界内，多端开发才能真正提高效率。
